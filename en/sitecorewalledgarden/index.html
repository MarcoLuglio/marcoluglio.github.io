Getting ready

Before you even think on doing anything, you will need a Sitecore license. It is what it is. If you do not work for a company that already has a deal with Sitecore, you may request a trial which will likely be followed up close by the Sitecore sales team.

Sitecore uses the .Net Framework, which is only available for Windows, and requires IIS for hosting. Sitecore 10 does have some parts written in .Net core, and an official Docker image, but because it is still bound to IIS, those are fat ass Windows images that take a lot of disk space, and require 32GB of RAM.

The Docker + Sitecore 10 path is easier to get started if you have the hardware. As simple as installing Git, Docker Desktop and running init.ps1 a command in PowerShell. https://github.com/Sitecore/docker-examples/tree/develop/getting-started and https://github.com/Sitecore/Helix.Examples/tree/master/examples/helix-basic-aspnetcore are two places where you can clone starting reference projects.

If you do not have 32GB of RAM, or just want a lighter environment, installing Sitecore on premisses is the way to go. For that you won‚Äôt need anything out of the ordinary:

  • .Net Framework (likely installed already)
  • IIS (enable it in Windows features)
  • SQL Server (use mixed mode authentication and write down the sa password, you will need it later)

Same for development:

  • Git
  • Visual Studio
  • SQL Server Management Studio*
  • A content serialization tool for Sitecore. I‚Äôll use Unicorn. That‚Äôs just a Nuget package, unlike other alternatives.

* It is not usual to access the database directly, but some questions about it may arise in the certification exam.

And if you are really committed to learn Sitecore, dotPeek is also your friend.

// step 1 - get index var Index = ContentSearchManager.GetIndex((SitecoreIndexableItem)Item); // or var Index = ContentSearchManager.GetIndex("sitecore_custom_index"); // step 2 - get context using(var context = Index.CreateSearchContext()) { // setp 3 - perform search var searchResults = context .GetQueryable<SearchResultItem>() .Where(s => s.Content.Contains("hello, world!")); // or var mySearch = context .GetQueryable<SearchResultItem>() .Where(x => x.Content.Contains("hello, world!")) .ToList() .Select(item => item.GetItem()).ToList(); }

Three of the primary indexes that come out of the box with a Sitecore installation are Master, Web, and Core.

Some of the reasons you may want to create a custom index are:

  • You want a certain part of the content tree to be isolated inside an index by itself.
  • You want to bring external data into Sitecore so it is searchable within Sitecore.
  • You need to shard indexes to support large amounts of content.
  • You want to define your custom start path so you do not have to index the entire content tree.

Caching

Sitecore uses four layers of caching.

  • HTML (default size is 200MB)
  • Item - Item cache for each database. Item caches have objects in the Sitecore class Sitecore.Data.Items.Item. Uses memoization. (default size is 1GB)
  • Data - Does not pull the item information until requested. (default size is 1GB)
  • Prefetch - Loads all of the frequently-loaded data up front. (default size is 1GB)

Update the file using the example below and check your changes visiting /sitecore/admin/showconfig.aspx, then search for patch:source="custom.prefetch.config".

IIS root\yoursite\App_Config\Include\custom.caching.config

<configuration xmlns:patch="http://www.sitecore.net/xmlconfig" xmlns:set="http://www.sitecore.net/xmlconfig/set" xmlns:role="http://www.sitecore.net/xmlconfig/role" > <sitecore> <databases> <database id="web"> <cacheSizes role:require="ContentDelivery or Standalone"> <!-- item cache --> <items>1000MB</items> <!-- data cache --> <data>1000MB</data> </cacheSizes> <!-- prefetch cache --> <dataProviders> <dataProvider param1="$(id)"> <prefetch hint="raw:AddPrefetch"> <cacheSize>1000MB</cacheSize> </prefetch> </dataProvider> </dataProviders> </database> <database id="master"> <cacheSizes role:require="ContentManagement or Standalone"> <!-- item cache --> <items>1000MB</items> <!-- data cache --> <data>1000MB</data> </cacheSizes> <!-- prefetch cache --> <dataProviders> <dataProvider param1="$(id)"> <prefetch hint="raw:AddPrefetch"> <cacheSize>1000MB</cacheSize> </prefetch> </dataProvider> </dataProviders> </database> </databases> </sitecore> </configuration>

Custom pages for HTTP codes

To avoid exposing yellow screen of death and alike.

For 500, do as an ordinary .Net project. Create a static html page and register a custom error in the waeb.config file.

For other errors, need to create a pipeline.

NotFoundHandler.cs

using Sitecore.IO; using Sitecore.Pipelines.HttpRequest; namespace Redirects { public class NotFoundHandler { public void Process(HttpRequestArgs args) { if (Sitecore.Context.Site == null) { return; } if (Sitecore.Context.Item != null) { return; } var notFoundPath = Sitecore.Context.Site.Properties["notFoundItem"]; if (string.IsNullOrEmpty(notFoundPath)) { return; } var sitePath = Sitecore.Context.Site.StartPath; var path = FileUtil.MakePath(sitePath, notFoundPath); var notFoundItem = Sitecore.Context.Database.GetItem(path); if (notFoundItem != null) { Sitecore.Context.Item = notFoundItem; args.HttpContext.Response.StatusCode = 404; } } } }

<?xml version="1.0"?> <configuration xmlns:patch="http://www.sitecore.net/xmlconfig/"> <sitecore> <pipelines> <httpRequestBegin> <processor type="Redirects.NotFoundHandler, Redirects" patch:after="processor[@type='Sitecore.Pipelines.HttpRequest.ItemResolver, Sitecore.Kernel']" /> </httpRequestBegin> </pipelines> </sitecore> </configuration>
